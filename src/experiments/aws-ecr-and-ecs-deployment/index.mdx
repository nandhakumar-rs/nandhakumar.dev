<ExperimentOverview>

This experiment documents how to dockerize and deploy a simple conversational LLM chat application built with Next.js and FastAPI to AWS using Amazon ECR and Amazon ECS.

<ExperimentTimeline>

 <DeploymentBlock title="Deployment: ECR + ECS">
  After v7, the application was deployed to AWS.

  - Built the backend Docker image separately.\n
    Backend and frontend are different apps with different runtimes, ports, dependencies, logs, and scaling needs.

  - Built the frontend Docker image separately.
    This lets us update or redeploy the frontend without touching the backend.

  - Built images for `linux/amd64`.
    Mac M2 builds `arm64` images by default, but our ECS Fargate task was configured for `Linux/X86_64`.

  - Created an ECR repository for the backend.
    ECS needs an AWS-accessible registry to pull the backend image from.

  - Pushed the backend Docker image to ECR.
    This makes the backend image available for ECS deployment.

  - Created an ECR repository for the frontend.
    Keeps frontend and backend images as separate deployable artifacts.

  - Pushed the frontend Docker image to ECR.
    This makes the frontend image available for ECS deployment.

  - Created an ECS cluster.
    Acts as the logical workspace where frontend and backend services run.

  - Used the same ECS cluster for frontend and backend.
    Both services belong to the same application environment.

  - Created a backend ECS task definition.
    Defines backend image, CPU, memory, port `8000`, environment variables, secrets, and logs.

  - Created a frontend ECS task definition.
    Defines frontend image, CPU, memory, port `3000`, and logs.

  - Stored `OPENAI_API_KEY` in SSM Parameter Store.
    Keeps the secret outside code, Git, Docker images, and plain task definition variables.

  - Added SSM read permission to the ECS execution role.
    Allows ECS to inject the OpenAI API key into the backend container at runtime.

  - Created CloudWatch log groups.
    Stores backend and frontend container logs for debugging.

  - Created an Application Load Balancer.
    Provides one public URL for the deployed application.

  - Created a backend target group.
    Routes backend API traffic to the FastAPI ECS task.

  - Created a frontend target group.
    Routes frontend page traffic to the Next.js ECS task.

  - Created security groups.
    Allows public traffic to the ALB and allows only the ALB to reach ECS containers.

  - Created the backend ECS service.
    Keeps the FastAPI backend container running.

  - Created ALB listener rules for backend paths.
    Routes API paths like `/chat*`, `/health`, or preferably `/api/*` to the backend target group.

  - Made the frontend target group the default ALB route.
    Routes normal browser page requests to the Next.js frontend.

  - Created the frontend ECS service.
    Keeps the Next.js frontend container running.

  - Used the same ALB for frontend and backend.
    Gives both apps the same base URL and avoids CORS issues.

  - Tested the backend API.
    Verified `/health` and `/chat/stream` through the public ALB URL.

  - Tested the frontend app.
    Opened the ALB URL in the browser and confirmed the frontend can call the backend.

  - Added start and stop commands.
    Set ECS service `desired-count` to `0` when not using the app to stop Fargate compute cost.

  - Ready to use as a deployed LLM chat experiment.
